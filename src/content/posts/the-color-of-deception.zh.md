在基于计算机视觉的自动化世界里，我们常默认「像素就是像素」。但在维护拥有 13 万+ 下载的 MaaEnd 时，我得到了一个教训：**同一张图在不同硬件上可以完全不同**。

以下是两个「色彩谜题」如何逼我们重构整套识别策略的故事。

---

## 上篇：HDR 幽灵（0.65 相似度之谜）

一切始于一堆 GitHub issue。用户反馈 `templateMatch`（基于 OpenCV 的模板匹配）在固定、静态场景下失效。日志显示相似度只有 0.65，而我们的稳定阈值通常在 0.85+。

肉眼看去，用户的截图和我们的模板几乎一模一样。但像素级分析显示：**RGB 发生了明显偏移**。

### 元凶：Windows HDR 与色彩管理

HDR 扩大了动态范围，也改变了通过标准 API 捕获的 RGB 值。我们不可能为每款 HDR 显示器都做一套模板，于是实施了务实的**环境守卫**：

* **标准化模板**：所有模板统一为 Native SDR；
* **预检**：在运行前检测 HDR 或专用色彩配置是否开启，并以 UI 提示建议用户关闭，以保证最佳稳定性。

---

## 下篇：GPU 共谋（稳定的 +12 偏移）

正当我们以为色彩问题解决时，一个更隐蔽的 bug 出现了：关闭 HDR 的用户仍然通不过 `colourMatch` 校验。

这次的偏移极小——肉眼几乎无法分辨。我们发现，在同一「绿色」UI 元素上，某位用户的绿色通道始终比我们高 12 个单位。

交叉测试团队硬件后，我们得出一个结论：**Nvidia、AMD、Intel 显卡在驱动层对 Gamma 和对比度的处理并不一致**。即便设置相同，实际渲染的 RGB 也会因驱动显卡不同而有轻微差异。

### 两难：精度 vs 鲁棒性

我们面临选择：

* **放宽 RGB 阈值？** 会带来误触和误点；
* **Hook 渲染管线？** 获取原始 buffer 复杂且依赖平台，还可能被反作弊标记。

### 方案：拥抱 HSV 空间

我们意识到，Gamma 和对比度变化会改变颜色的「明度」和「饱和度」（RGB），但很少改变其「本质」（色相 Hue）。绿色按钮依然是绿色，只是饱和略高或略低。

我们将识别引擎重构为**混合多空间模型**：

* **以色相为核心的 HSV 匹配**：对彩色 UI 元素，从 RGB 切换到 HSV，对 H（色相）严格锁定，S、V 放宽以吸收 GPU 差异；
* **RGB 兜底**：对黑白灰等 Hue 无定义或不稳定的元素，仍用优化过阈值的 RGB 匹配。

## 工程心得

这段经历说明：**鲁棒软件必须考虑物理层**。从「固定像素」思维转向「感知色彩」思维后，我们消除了整类硬件相关 bug，显著提升了 MaaEnd 在数千设备上的稳定性。
