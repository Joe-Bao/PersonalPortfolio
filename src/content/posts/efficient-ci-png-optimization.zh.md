在 MaaEnd 这个基于计算机视觉的项目中，我们的「事实来源」是数千张用于 `templateMatch` 的图片模板。随着项目增长，我们遇到了经典的 DevOps 困境：测试套件变得臃肿、缓慢，在 GitHub Actions 上运行既耗时又耗配额。

---

## 挑战：臃肿的模板与设备差异

自动化逻辑必须在数千种不同设备上保持稳健。为此我们做了大量「跨节点校验」——确保模板只触发预期节点，忽略其余。

然而这些模板多为全屏截图（即使是 720p），仓库体积开始暴涨。我们必须在**无损**的前提下压缩图片，任何有损压缩都可能破坏 OpenCV 的匹配精度。

---

## 第一步：用 oxipng 做无损压缩

我用 `oxipng`（多线程无损 PNG 优化器）实现了 CI 流水线。为保证数据绝对一致，增加了**像素级完整性校验**：

1. 将原图和优化后的图转为灰度；
2. 逐像素比对；
3. 仅在两图数学意义上完全相同时才提交变更。

---

## 第二步：瓶颈

第一次跑直接炸了。在标准 GitHub Action 环境中优化数千张高清模板，预估耗时超过 **10 小时**，足以卡住整个开发节奏并耗尽 CI 配额。

---

## 第三步：「增量」方案

我立刻转向**增量优化策略**：

- **本地预处理**：在本地批量完成首轮优化，为仓库打底；
- **基于 Hash 的跟踪**：用 JSON 维护每张优化后图片的 SHA-256；
- **智能 CI 逻辑**：GitHub Action 优先查 Hash，若未变则跳过优化。

---

## 结果

加入增量缓存后，流水线在 3 分钟内跑完：

![PNG 图片优化 CI 运行](/blog-ci-optimization.png)

效果：

- **CI 耗时**：从理论 10+ 小时压到每次 PR 约 **2 分钟**；
- **存储效率**：单张图片平均节省约 **10%** 体积，且识别精度不变；
- **开发体验**：新模板由 GitHub Actions 机器人自动优化并推送回分支，无需人工介入。
